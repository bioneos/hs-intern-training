Version Management and GitHub

A beginner module for high school interns learning how Git and GitHub track project history, support teamwork, and protect shared code.

Learning Goals

  1. Explain what Git and GitHub do.
  2. Describe repositories, commits, branches, forks, and pull requests.
  3. Use the basic Git workflow: status, add, commit, pull, and push.
  4. Understand the BioNeos feature-branch workflow.

Git vs. GitHub

Git

Version-control software installed on your computer. It tracks changes and saves checkpoints called commits.

  • Version-control software installed on a computer
  • Tracks changes made to project files
  • Creates commits that save snapshots of work
  • Supports branches and merging
  • Can be used without an internet connection
  • Works through command-line tools or desktop applications
  • Stores repository history locally

GitHub

A web platform that hosts Git repositories so teams can share, review, and collaborate on code.

  • An online platform that hosts Git repositories
  • Makes repositories accessible through the internet
  • Supports collaboration between developers
  • Adds pull requests and code-review tools
  • Includes issues, projects, actions, and team-management features
  • Provides backups and remote access to repositories
  • Can publish websites through GitHub Pages

Git can be used by itself, but GitHub makes collaboration much easier.

Why We Use Git

Track

Record

Save meaningful checkpoints with commit messages.

Restore

Return to earlier versions when something breaks.

Experiment

Try ideas without damaging the shared project.

Analyze

Review history to understand how a project developed.

Merge

Combine work from multiple contributors.

Why We Use GitHub

Share

Store repositories online so team members can access the project.

Collaborate

Allow multiple developers to contribute to the same project.

Review

Use pull requests to discuss, review, and approve changes.

Organize

Track bugs, tasks, and ideas with Issues and Projects.

Back Up

Keep a remote copy of the repository and its history.

Publish

Share websites and project documentation with GitHub Pages.

Use the down arrow to explore different git commands.

git init

Start a New Repository

Initializes a new Git repository in the current folder. 

git init

Use this when a project folder is not already connected to Git.

git clone

Copy an Existing Repository

Downloads a remote repository and creates a local copy on your computer. 

git clone https://github.com/username/repository.git

Use this when the project already exists on GitHub or another remote service.

git status

Check the Current State

Shows which files have changed, which files are staged, and which files are untracked. 

git status

Run this frequently so you know what Git sees before committing.

git add

Stage Changes

Selects changes that will be included in the next commit. 

git add filename.html
git add .

The first command stages one file. The second stages all current changes.

git commit

Save a Checkpoint

Creates a new commit containing the changes that have been staged. 

git commit -m "Add hydration dashboard"

A good commit message briefly explains what changed.

git log

Review Project History

Displays previous commits, including messages, authors, dates, and identifiers. 

git log
git log --oneline

The --oneline option creates a shorter, easier-to-scan history.

git branch

Create or View Branches

Branches allow developers to work on different versions or features without immediately changing the main project. 

git branch
git branch new-feature

The first command lists branches. The second creates a new branch.

git checkout

Move to Another Branch

Switches the working folder to a specified branch or commit. 

git checkout new-feature

A newer alternative for switching branches is: 

git switch new-feature

git merge

Combine Branches

Brings changes from one branch into the branch you are currently using. 

git switch main
git merge new-feature

Switch to the destination branch before running the merge.

git pull

Download Remote Changes

Downloads new changes from the remote repository and merges them into the current local branch. 

git pull

Pull before beginning work so your local project is up to date.

git push

Upload Local Commits

Sends committed changes from the local repository to the remote repository. 

git push

Only committed changes are uploaded. Uncommitted edits remain on your computer.

git status

Check Your Working Directory

Shows which files have changed, which files are staged, and which files are untracked. 

git status

Run this frequently to understand the current state of your repository.

git log

View Commit History

Displays previous commits, including commit messages, authors, dates, and commit hashes. 

git log
git log --oneline

The --oneline option creates a shorter, easier-to-read history.

git blame

See Who Changed Each Line

Shows the commit hash, author, timestamp, and content associated with each line in a file. 

git blame filename.js

Use this to understand when and why a particular line was changed.

git restore

Discard Uncommitted Changes

Restores a file in your working directory to its last committed version. 

git restore filename.js

This permanently removes uncommitted changes from the selected file.

git revert

Safely Undo a Commit

Creates a new commit that reverses the changes from an earlier commit without rewriting project history. 

git revert COMMIT_HASH

This is usually the safest option for undoing work that has already been shared.

git reset

Move the Branch and HEAD

Moves the current branch and HEAD pointer to a selected commit. 

git reset --mixed COMMIT_HASH

Soft

Keeps changes staged.

git reset --soft COMMIT_HASH

Mixed

Keeps changes but removes them from staging.

git reset --mixed COMMIT_HASH

Hard

Deletes later commits and working-directory changes.

git reset --hard COMMIT_HASH

GitHub Repositories

A repository, or repo, is a project folder tracked by Git. GitHub is a online platform for creating, managing, and sharing repositories. It uses Git to provide distributed version control, which is form of version control in which the complete codebase, including its full history, is mirrored on every developer's computer.

The repo is held in 2 places.

Local Repo

The copy on your computer where you edit and test files.

Remote Repo

The online copy hosted on GitHub for sharing and collaboration.

What Is Inside a Repo?

  • Source code files
  • Images and other assets
  • Documentation, often including a README.md
  • Commit history
  • Branches
  • Issues and pull requests

Forks and Branches

Forks and branches are 2 ways for developers to experiment without immediately changing the main project. Both make copies of repos for developer use but do it in importantly different ways.

Branch

An isolated workspace inside an existing Git repository.

Fork

A separate copy of a repository stored under another GitHub account.

Use the down arrow to explore how each one works.

What Is a Branch?

Work Inside the Repository

A branch creates a separate line of development within the same repository.

  • starts from an existing commit
  • keeps unfinished work away from the main branch
  • can be merged back into another branch
  • is commonly used for features, bug fixes, and experiments

Branches are normally used by people who already have permission to contribute to the repository.

Branch Workflow

1. Create

git switch -c feature-name

Create and enter a new branch.

2. Develop

git add .
git commit -m "Add feature"

Stage and commit the work.

3. Share

git push -u origin feature-name

Upload the branch to GitHub.

A pull request can then be opened to review and merge the branch into the main project.

Branches Share One Repository

Main Branch → Feature Branch → Commits → Pull Request → Merge

Same Repository

The main branch and feature branch belong to the same project.

Shared Access

Repository collaborators can view and work with the branch.

What Is a Fork?

Copy the Entire Repository

A fork creates a separate GitHub repository under your own account.

  • does not require write access to the original repository
  • can contain its own branches and commits
  • can be changed without affecting the original project
  • can be used as the starting point for a different project

Forking is a GitHub feature. Git itself does not include a git fork command.

Fork Workflow

How the Repositories Connect

  1. Fork the original repository on GitHub.
  2. Clone your fork to your local computer.
  3. Create a branch and make changes locally.
  4. Push the changes to your GitHub fork.
  5. Open a pull request to the original repository.

A fork is a separate GitHub repository, while a clone is the copy stored on your local computer.

Keeping a Fork Updated

The original repository is commonly added as a second remote named upstream.

Add the Original Repository

git remote add upstream ORIGINAL-REPOSITORY-URL

Download New Changes

git fetch upstream
git switch main
git merge upstream/main

This keeps your fork synchronized with changes made to the original project.

Branch or Fork?

Use a Branch When...

  • you have write access
  • you are part of the project team
  • the work belongs in the same repository
  • you are developing a feature or fixing a bug

Use a Fork When...

  • you do not have write access
  • you are contributing to another organization’s project
  • you want an independent copy
  • you are contributing to open-source software

Quick Comparison

Location

Branch: Inside the original repository.

Fork: A separate repository.

Permission

Branch: Usually requires write access.

Fork: Does not require write access.

Best Use

Branch: Internal development.

Fork: External or open-source contributions.

Which Should You Use?

Scenario A

You are a member of the development team and need to add a dashboard feature.

Use a branch.

Scenario B

You found an open-source project and want to propose a fix without having write access.

Use a fork.

Remember

Branch

A separate workspace inside the same repository.

Fork

A separate copy of the repository under another account.

Branches separate the work. Forks separate the repository.
